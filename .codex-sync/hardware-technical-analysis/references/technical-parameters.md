# 硬件技术分析参考框架

本文件用于给智能硬件整机技术分析提供统一的拆解路径、整机框图视角、运行流程视角、参数维度、公式类别和工程判断视角。不要把这里的典型参数或常见方案当作定值；它们只用于建立分析框架和粗估逻辑。

## 1. 使用方式

遇到硬件对象时，不要直接把本文件整段复述给用户。正确用法是：

1. 先确定分析对象属于哪一类
2. 再挑选对应链路和参数维度
3. 最后把参数翻译成工程结论

优先回答以下问题：

- 整机由哪些模块构成
- 系统是如何从上电一路运行到输出结果的
- 每个模块分别负责什么，模块之间如何协作
- 哪个模块决定了核心性能
- 哪个模块决定了主要成本
- 哪个模块决定了可靠性和量产风险
- 哪个模块最可能成为竞品差异来源

## 2. 证据与结论分层

输出时建议把判断分成三层：

| 层级 | 含义 | 典型来源 |
|---|---|---|
| 已确认事实 | 可以直接引用或观察到 | 规格书、认证、原理图、实测、拆机图 |
| 工程推断 | 由事实和常识推导出的高概率判断 | 参数组合、架构模式、行业通用做法 |
| 待验证假设 | 证据不足、但值得提出的方向 | 型号猜测、工艺猜测、成本估算 |

经验规则：

- 没有直接证据时，不要给出具体芯片型号
- 没有测试条件时，不要把参数横向硬比较
- 没有结构细节时，不要把精度问题简单归因于传感器

## 3. 输出介质建议

针对智能硬件整机分析，建议默认按三层组织产物：

| 介质 | 角色 | 主要内容 |
|---|---|---|
| Markdown | 技术底稿 | 事实、参数、证据、结论、风险、表格 |
| HTML | 展示成品 | 分层阅读、动画解释、评审展示、交互浏览 |
| SVG | 图示资产 | 整机框图、流程图、链路图、核心原理图 |

选择规则：

- 需要编辑、复核、继续追写时，优先保留 Markdown
- 需要展示、汇报、演示、网页化浏览时，补 HTML
- 需要高清图、放大、局部高亮、动画时，优先用 SVG

建议做法：

- 先写 Markdown，再从 Markdown 抽取结构生成 HTML
- 图示尽量独立成 SVG 文件，不要只截图嵌入
- 若做动画，让 SVG 或 HTML 承担动画层，Markdown 不负责动画

## 4. 整机框图拆解法

分析智能硬件整机时，优先重建模块框图，而不是先钻到单芯片。

建议先按下列模块层建立框图：

| 模块层 | 典型内容 | 主要职责 |
|---|---|---|
| 电源层 | 充电、保护、PMIC、电池、稳压 | 给各模块供电，决定续航和峰值能力 |
| 感知层 | 传感器、按键、旋钮、麦克风、摄像头等 | 采集物理量或用户输入 |
| 计算层 | MCU、SoC、DSP、FPGA、存储 | 处理数据、做控制和任务调度 |
| 控制与执行层 | 电机、阀、加热器、继电器、蜂鸣器等 | 把算法结果转成物理动作 |
| 交互层 | 屏幕、LED、语音、按键反馈 | 向用户呈现状态和结果 |
| 通信层 | BLE、Wi-Fi、USB、CAN、NFC 等 | 对外连接、同步、升级 |
| 云与应用层 | App、云端、后台服务 | 扩展配置、数据管理、远程能力 |
| 结构与热层 | 外壳、支架、导热、防水、减振 | 保障机械、环境和热稳定性 |

框图至少回答：

- 哪些模块是主链路模块
- 哪些模块是支撑模块
- 哪些模块直接影响核心指标
- 哪些模块可平台复用

如果用户没有给图，也应输出文字框图，例如：

`电池/USB -> PMIC -> MCU -> 传感器AFE -> 算法处理 -> 显示/无线输出`

输出到 SVG 时，至少保证：

- 模块层级清晰
- 主链路和辅助链路可区分
- 箭头方向清楚
- 模块命名与正文一致
- 放大后仍可读

## 5. 整机运行流程拆解法

整机分析默认要重建系统如何跑起来。

建议按阶段描述：

1. 上电 / 唤醒
2. 自检 / 传感器初始化 / 参数加载
3. 待机监听或连续采样
4. 数据采集与预处理
5. 状态判断 / 控制决策 / 算法计算
6. 执行动作或结果显示
7. 数据上报 / 同步 / 存储
8. 休眠、异常恢复或保护动作

每个阶段都应说明：

- 由哪个模块主导
- 调用了哪些外围模块
- 输入是什么
- 输出是什么
- 失败时会发生什么

如果设备是闭环系统，还要说明：

- 控制周期有多长
- 反馈从哪里回来
- 哪些阶段影响稳定性与响应速度

如果输出为 HTML / SVG 演示，还应考虑：

- 哪些阶段适合顺序点亮
- 哪些模块切换适合做高亮动画
- 哪些箭头表示数据流，哪些箭头表示能量流或控制流

## 6. 系统拆解五条链路

硬件技术分析优先沿五条链路展开：

### 3.1 能量链

`输入电源 -> 转换/稳压 -> 储能 -> 负载 -> 峰值/平均功耗 -> 热`

看什么：

- 供电方式是否匹配场景
- 峰值电流是否被低估
- 电源噪声是否会污染信号链
- 发热是否反过来影响精度、寿命或舒适性

### 3.2 信号链

`传感器/输入 -> 激励/调理 -> AFE/ADC -> MCU/SoC -> 算法 -> 显示/输出`

看什么：

- 最弱的信噪比环节在哪里
- 量化误差、漂移、带宽、滤波如何分配
- 精度是硬件决定还是算法补偿决定

### 3.3 控制链

`采样 -> 状态估计 -> 控制律 -> 驱动 -> 执行机构 -> 反馈`

看什么：

- 闭环还是开环
- 主要延迟来自采样、计算还是执行
- 稳定性、超调、响应时间由什么限制

### 3.4 数据链

`本地存储/总线 -> 无线/有线传输 -> 主机/App/云端`

看什么：

- 带宽与时延是否匹配业务
- 连接稳定性与协议兼容性如何
- 数据完整性、升级能力、安全性是否是薄弱点

### 3.5 结构链

`外力/热/液体/灰尘/振动 -> 外壳/支架/界面材料/密封 -> 核心功能模块`

看什么：

- 力传递和热传递是否可控
- 结构公差是否侵蚀性能一致性
- 装配与维修是否制造隐性成本

## 7. 通用性能参数

优先确认这些基础维度：

| 参数 | 说明 |
|---|---|
| 量程 / 工作范围 | 设备可覆盖的输入或输出范围 |
| 精度 | 与真实值偏差的上限或统计值 |
| 分辨率 | 最小可区分变化量 |
| 重复性 | 多次测试结果的一致性 |
| 线性度 | 输出相对理想模型的偏离 |
| 灵敏度 | 输入变化对应输出变化幅度 |
| 响应时间 | 从输入变化到稳定输出的时间 |
| 带宽 / 采样率 / 刷新率 | 动态能力与可观察性 |
| 功耗 | 平均、峰值、待机功耗 |
| 可靠性 | 漂移、寿命、失效率、环境适应性 |

分析提醒：

- “高分辨率”不等于“高精度”
- “高采样率”不等于“有效信息更多”
- “规格上限”与“长期稳定兑现”通常不是一回事

## 8. 传感器与模拟前端

分析传感器或测量模块时重点看：

| 维度 | 说明 |
|---|---|
| 物理原理 | 电阻式、电容式、压电式、磁式、光学式、超声式等 |
| 灵敏度 | 输入变化对输出的增益 |
| 噪声 | 本底噪声、量化噪声、环境噪声 |
| 温漂 | 温度变化导致的零点或增益变化 |
| 零点漂移 | 长时间运行后的基线偏移 |
| AFE 需求 | 是否需要激励、放大、滤波、ADC |
| 标定需求 | 出厂标定、温补、现场校准 |
| 安装应力 | 装配力、胶水、螺钉预紧是否影响测量 |

高频误判：

- 只看传感器规格，不看安装结构和热环境
- 只看 ADC 位数，不看有效位数和前端噪声
- 只看静态精度，不看动态工况与漂移

## 9. 处理器、控制器与执行机构

| 维度 | 说明 |
|---|---|
| MCU / SoC / DSP / FPGA 类型 | 决定算力、实时性和开发复杂度 |
| 主频与算力 | 决定控制周期、算法复杂度、UI/通信余量 |
| 存储资源 | Flash、SRAM、外部存储容量 |
| 外设接口 | ADC、PWM、I2C、SPI、UART、CAN、USB 等 |
| 驱动能力 | 马达、加热器、阀、继电器等执行能力 |
| 实时性 | 是否需要硬实时调度 |
| 升级机制 | Bootloader、OTA、安全回滚 |
| 保护策略 | 看门狗、限流、过温、失效安全状态 |

若对象带执行机构，还要额外判断：

- 负载特性是否清楚
- 峰值驱动是否被电源或热限制
- 反馈闭环是否足够快且足够稳

如果是整机主控，还要额外问：

- 主控承担的是实时控制、协议桥接还是应用逻辑
- 时间关键任务与非关键任务是否隔离
- 主控是否同时承载 UI、无线和控制，导致资源竞争

## 10. 电源与功耗

| 维度 | 说明 |
|---|---|
| 供电方式 | 电池、USB、适配器、PoE、能量采集 |
| 电池体系 | 锂电、一次性电池、超级电容等 |
| 电源拓扑 | LDO、Buck、Boost、Buck-Boost、隔离电源 |
| 峰值电流 | 无线发射、驱动、电机、加热等场景尤其关键 |
| 待机策略 | 休眠、唤醒、占空比控制 |
| 纹波与噪声 | 是否影响模拟前端、RF 或精密测量 |
| 续航估算 | 电池容量 / 平均电流，只能粗估 |

分析提醒：

- 续航差往往不是电池小，而是峰值场景和待机策略设计差
- 模拟前端和无线发射共享电源时，噪声隔离尤其关键

## 11. 通信与连接

| 维度 | 说明 |
|---|---|
| 协议 | BLE、Wi-Fi、Zigbee、Thread、LoRa、NFC、USB、CAN 等 |
| 带宽 | 数据吞吐是否匹配业务需求 |
| 时延 | 控制类场景通常比监测类更敏感 |
| 抗干扰 | 金属壳体、距离、共存干扰、布线和天线布局 |
| 兼容性 | 协议栈、主机、App、OS 支持情况 |
| 功耗耦合 | 通信策略对续航与热的影响 |

通信问题通常不是“连不上”这么简单，还要问：

- 在最差环境下是否稳定
- 配网或配对成本是否过高
- 固件升级是否依赖通信链且缺少兜底

对智能硬件整机，还要判断通信层在整机中的角色：

- 只是配网和配置
- 持续传数据
- 承担远程控制
- 承担 OTA 与设备运维

## 12. 机械结构、热管理与材料

| 维度 | 说明 |
|---|---|
| 承载路径 | 力、热、振动如何在结构内传递 |
| 刚度与强度 | 决定精度、寿命和安全裕量 |
| 材料选择 | 铝、不锈钢、工程塑料、复材、硅胶等 |
| 公差与装配 | 影响一致性、校准量和良率 |
| 密封与防护 | IP、防尘、防水、防腐蚀 |
| 散热 | 自然对流、热界面材料、金属壳散热 |
| 维修性 | 可拆卸性、返修难度、胶粘与焊接策略 |

高精度硬件常见规律：

- 精度不只取决于传感器，也取决于承载结构和装配一致性
- 热设计不只关乎安全，也关乎漂移和寿命
- 胶水、螺钉、预紧、焊接顺序都可能改变最终指标

## 13. 算法、标定与信号处理

常见链路：

`采样 -> 去噪 -> 校准 -> 补偿 -> 特征提取/控制 -> 输出`

| 类型 | 关注点 |
|---|---|
| 均值 / 中值滤波 | 抑制随机噪声和毛刺 |
| IIR / FIR | 频率响应、延迟、计算量 |
| 卡尔曼 / 互补滤波 | 多源融合、动态跟踪、噪声建模 |
| 温度补偿 | 多项式、查表、分段模型 |
| 标定算法 | 零点、斜率、多点拟合、非线性修正 |
| 控制算法 | PID、状态机、前馈/反馈、异常处理 |

分析算法时至少问：

- 它补偿的是随机误差、系统误差还是装配差异
- 算法是否在掩盖硬件不足
- 算法带来的延迟是否会反噬体验或控制稳定性

对整机运行流程，还应回答：

- 算法在本地实时运行还是离线 / 云端运行
- 哪些参数在出厂阶段确定，哪些可在运行时自适应
- 异常检测和保护逻辑位于哪一层

## 14. 可靠性与环境适应性

| 维度 | 说明 |
|---|---|
| 工作温度 | 常温外是否仍满足指标 |
| 防水防尘 | IP 等级与测试条件 |
| 抗跌落 / 振动 | 运输和使用冲击能力 |
| 老化与疲劳 | 机械件、胶水、电池、传感器漂移 |
| ESD / EMC | 静电和电磁环境对稳定性的影响 |
| 容错能力 | 掉电、断连、异常输入时是否安全退化 |

常见失效模式：

- 漂移超标
- 胶粘或焊点失效
- 连接器松脱
- 电池衰减
- 进水腐蚀
- 执行机构卡滞
- 固件异常导致假死或数据错误

整机级可靠性还要检查：

- 单模块失效会不会拖垮整机
- 是否有降级运行模式
- 校准信息、配置数据和升级状态是否可靠保存

## 15. 成本、制造与供应链

分析成本时至少拆成：

| 维度 | 内容 |
|---|---|
| 电子 BOM | 芯片、被动件、传感器、电池、连接器 |
| 结构 BOM | 外壳、支架、密封件、紧固件、装饰件 |
| 制造成本 | SMT、组装、老化、标定、测试、包装 |
| 开发成本 | 固件、算法、App、治具、认证 |
| 供应风险 | 单一来源、交期、停产、认证限制 |

降本分析重点：

- 哪些器件是“显性贵”，哪些工序是“隐性贵”
- 降本是否会转移到校准、返修、售后或良率损失
- 竞品低价是靠平台复用、器件替代，还是规格收缩

整机成本分析时，优先区分：

- 决定核心功能的模块成本
- 决定用户体验的模块成本
- 决定量产复杂度的工序成本
- 看似便宜但会放大售后风险的降本点

## 16. 常见公式类别

以下公式用于粗估和解释，不应替代正式设计计算：

- 电桥与应变：`ΔR / R = K × ε`
- 功率：`P = V × I`
- 旋转机械功率：`P = T × ω`
- 采样量化：`LSB = 满量程 / 2^N`
- RC 截止频率：`f_c = 1 / (2πRC)`
- 电池续航粗估：`续航 ≈ 电池容量 / 平均电流`
- 热阻粗估：`ΔT ≈ P × Rθ`

使用这些公式时要写清楚：

- 近似条件
- 输入参数来源
- 误差边界和不适用情形

## 17. 分析提醒

- 参数必须带单位，必要时带测试条件
- 整机分析优先给出框图、运行流程和模块职责
- 不要把“功能列表”误写成“系统架构”
- 不要只说模块名称，要说模块之间怎么协作
- HTML 应作为展示层，不应成为唯一事实来源
- Markdown 与 HTML 的图名、模块名、结论应保持一致
- SVG 图示应使用可编辑矢量对象，不要退化成位图截图
- 竞品对比必须对齐量程、精度定义和使用场景
- 粗估 BOM 时不要忽略校准、测试、结构件和认证成本
- 不要把软件补偿后的指标直接等同于硬件原生能力
- 不要把实验室最佳值直接等同于量产一致性
- 当证据不足时，优先输出“最可能方案 + 依据 + 反例可能性”
