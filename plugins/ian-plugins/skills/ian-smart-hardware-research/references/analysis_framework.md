# 智能硬件用户调研分析框架

## 一、需求分析维度

### 1.1 功能性需求
- 核心功能期望（产品应该做什么）
- 辅助功能需求（增值功能期望）
- 性能期望（响应速度、准确率等）

### 1.2 情感性需求
- 安全感（使用放心、数据安全）
- 成就感（使用后的正向反馈）
- 社交需求（分享、展示）

### 1.3 使用场景分类
- **日常场景**：高频使用的基本情境
- **特殊场景**：特定时间/地点的使用
- **应急场景**：故障、意外情况下的使用

## 二、痛点分析维度

### 2.1 产品层面
- 硬件问题（质量、做工、耐用性）
- 软件问题（App、固件、连接）
- 功能缺失（期望有但实际没有）

### 2.2 体验层面
- 学习成本（上手难度）
- 操作复杂度（日常使用是否便捷）
- 维护成本（充电、清洁、维修）

### 2.3 服务层面
- 售前服务（咨询、体验）
- 售后支持（保修、客服、退换）
- 生态配套（配件、耗材、升级）

## 三、讨论热点识别

### 3.1 热点类型
- **功能争议**：某功能有用 vs 鸡肋
- **价格争议**：值不值这个价
- **品质争议**：质量参差不齐
- **对比讨论**：与竞品的比较

### 3.2 热度评估
- 🔥🔥🔥 高频讨论（多个平台大量提及）
- 🔥🔥 中等讨论（单一平台集中讨论）
- 🔥 低度讨论（零星提及）

## 四、分析方法论

### 4.1 内容分析法
1. 收集原始讨论内容
2. 编码分类（需求/痛点/好评）
3. 统计频次
4. 提炼主题

### 4.2 情感分析
- 正面情感（满意、惊喜、推荐）
- 负面情感（失望、抱怨、后悔）
- 中性情感（客观描述、询问）

### 4.3 用户分群
- 核心用户（高频使用、深度体验）
- 普通用户（日常使用、基本功能）
- 潜在用户（观望、犹豫、询问）

## 五、输出质量标准

### 5.1 数据可信度
- 至少3个平台的数据交叉
- 每个观点有具体用户原声支撑
- 区分个人观点和普遍现象

### 5.2 洞察深度
- 不仅罗列现象，更解释原因
- 识别未明说的潜在需求
- 提供可操作的产品建议
