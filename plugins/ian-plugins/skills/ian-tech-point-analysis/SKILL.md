# 技术点深度解析

## 使用方式

**本技能仅通过显式调用触发**。

正确调用方式：
- "使用 ian-tech-point-analysis 分析功率计的应变片测量原理"
- "用技术点分析技能深入解析快充技术的协议标准"
- "调用技术点解析技能，分析OIS光学防抖的机械结构"

## 技能概述

本技能针对用户指定的特定技术点进行深度技术分析，与整机分析不同，它聚焦于：
- 单一技术原理的深度解析
- 特定组件/传感器的工作机制
- 特定算法的实现逻辑
- 特定结构的设计原理

核心特点：
- **深度优先**：不求广度，追求技术细节的深度
- **证据驱动**：每个技术细节都需有资料支撑
- **可视化辅助**：结合图示、动画帮助理解
- **专业输出**：生成可供工程师参考的技术报告

## 与整机分析的区别

| 对比项 | ian-tech-point-analysis | ian-hardware-technical-analysis |
|--------|------------------------|--------------------------------|
| 分析范围 | 特定技术点（1-3个） | 整机全系统 |
| 分析深度 | 极深（原理级、公式级） | 中等（模块级、链路级） |
| 资料量 | 大量（围绕单点搜集） | 适中（系统级资料） |
| 输出重点 | 技术原理详解 | 系统架构全景 |
| 典型用户 | "应变片怎么测扭矩" | "功率计整机怎么工作" |

## 输出要求

### 必交产物
1. **Markdown 技术报告** (`report.md`) - 完整技术分析文档
2. **HTML 展示报告** (`report.html`) - 可视化版本，使用 `brutally minimal` 风格
3. **技术图示** - SVG/PNG 格式的原理图、结构图、流程图
4. **动画/交互演示** (可选) - 用于说明动态过程
5. **数据校验** - 必须调用 `fact-checker` 验证关键数据

### 输出流程
```
技术点确认 → 资料搜集 → 原理分析 → Markdown草稿 → 可视化制作 → fact-checker校验 → HTML成品
```

## 工作流程

### Phase 1: 明确技术点

确认分析对象：
- **技术点名称**：（如"应变片扭矩测量"）
- **所属领域**：（如"传感器技术"）
- **分析目标**：（如"理解测量精度如何保证"）
- **深度要求**：（原理级/实现级/公式级）

### Phase 2: 多维度资料搜集

**必须覆盖的资料类型：**

1. **学术论文/专利**
   - IEEE、ACM 等学术数据库
   - 相关技术专利（USPTO、CNIPA）
   - 博士/硕士论文

2. **技术规格书**
   - 传感器/芯片 datasheet
   - 接口协议规范
   - 行业标准文档

3. **工程实践资料**
   - 应用笔记 (Application Notes)
   - 设计参考 (Reference Designs)
   - 技术博客/开发者文档

4. **竞品/产品资料**
   - 同类产品技术白皮书
   - 拆解分析报告
   - 评测文章

**资料搜集策略：**
```
核心技术概念 → 相关标准/规范 → 实现方案对比 → 典型产品案例 → 常见问题与解决
```

### Phase 3: 技术原理深度分析

分析框架（根据技术点类型选择）：

#### 传感器类技术点

| 分析维度 | 内容 |
|---------|------|
| 物理原理 | 基于什么物理效应？公式是什么？ |
| 敏感元件 | 材料、结构、工艺 |
| 信号调理 | 放大、滤波、ADC 过程 |
| 误差来源 | 温漂、非线性、噪声等 |
| 校准方法 | 如何标定、补偿 |
| 性能边界 | 精度、量程、响应时间的理论极限 |

#### 算法类技术点

| 分析维度 | 内容 |
|---------|------|
| 问题建模 | 数学模型、约束条件 |
| 算法原理 | 核心思想、推导过程 |
| 实现步骤 | 伪代码、流程图 |
| 复杂度分析 | 时间/空间复杂度 |
| 参数影响 | 关键参数如何影响结果 |
| 优化方向 | 现有改进方案对比 |

#### 结构类技术点

| 分析维度 | 内容 |
|---------|------|
| 功能需求 | 结构需要满足什么功能 |
| 设计方案 | 可选方案对比、选型依据 |
| 力学分析 | 受力、形变、强度校核 |
| 材料选择 | 材料特性、选型理由 |
| 制造工艺 | 加工方法、公差控制 |
| 可靠性 | 疲劳、磨损、失效模式 |

### Phase 5: 可视化制作

根据技术点类型制作：

| 类型 | 可视化内容 |
|-----|-----------|
| 传感器 | 敏感元件结构图、信号链框图、误差曲线图 |
| 算法 | 流程图、收敛过程动画、复杂度对比图 |
| 结构 | 3D结构图、力传递路径图、形变仿真图 |
| 电路 | 原理图、时序图、频谱图 |

**动画制作原则：**
- 使用 HTML/CSS/SVG 实现轻量动画
- 展示动态过程（如信号流向、算法迭代）
- 可交互（如调节参数看结果变化）

### Phase 6: 数据校验

**必须调用 `fact-checker` 技能**，校验：
- [ ] 物理公式和数学推导的正确性
- [ ] 性能参数的来源和准确性
- [ ] 技术方案的时效性
- [ ] 引用资料的可信度

### Phase 7: HTML 报告生成

**调用 `frontend-design` 技能**：
```
Skill: frontend-design
要求：
- 风格: brutally minimal
- 内容: 基于 Markdown 技术报告
- 包含: 技术原理图解、公式渲染、对比表格
- 嵌入: SVG图示、动画演示
- 结构: 清晰的层级和导航
```

## 技能调用链

本技能可以调用：

| 场景 | 调用技能 | 目的 |
|------|---------|------|
| 需要整机上下文 | `ian-hardware-technical-analysis` | 了解技术点在整机中的位置和作用 |
| 需要用户需求数据 | `ian-smart-hardware-research` | 了解该技术点的用户关注焦点 |
| 需要竞品数据 | `ian-competitive-analysis` | 对比不同产品的技术实现 |
| 数据校验 | `fact-checker` | 验证技术数据的准确性 |
| 可视化 | `frontend-design` | 生成 HTML 技术报告 |

## 质量控制清单

```
□ 技术点定义清晰
□ 资料搜集全面（学术+工程+产品）
□ 原理分析深入（公式+机制+边界）
□ Markdown 报告结构完整
□ 技术图示已制作（至少3张）
□ fact-checker 已调用并修正错误
□ HTML 报告使用 brutally minimal 风格
□ 所有技术细节都有引用支撑
□ 参考资料完整标注
```

## 注意事项

1. **显式调用** - 本技能仅响应显式调用
2. **深度优先** - 不要追求面面俱到，要深入核心原理
3. **公式可验证** - 所有公式都要有推导或引用来源
4. **图表清晰** - 技术图示要标注清楚，避免歧义
5. **边界明确** - 明确说明技术的适用范围和局限性
6. **必须校验** - 所有关键数据必须经过 fact-checker 验证
7. **区分证据等级** - 明确区分已确认事实、工程推断、待验证假设
