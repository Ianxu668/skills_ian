# 智能硬件整机技术分析

## 使用方式

**本技能仅通过显式调用触发**。

正确调用方式：
- "使用 ian-hardware-technical-analysis 分析智能手表技术原理"
- "用智能硬件技术分析技能拆解智能音箱"
- "调用整机技术分析技能，分析扫地机器人架构"

## 技能概述

把智能硬件问题拆成可验证的整机工程问题，而不是停留在参数罗列或零散器件描述。优先重建整机框图、运行流程和模块职责，再下钻到器件、结构和算法细节。

## 输出要求

### 必交产物
1. **Markdown 报告** (`report.md`) - 技术底稿，包含完整数据和工程判断
2. **HTML 报告** (`report.html`) - 展示成品，使用 `brutally minimal` 风格
3. **SVG 图示** - 至少包含整机模块框图和运行流程图
4. **数据校验** - 必须调用 `fact-checker` 技能验证关键数据

### 输出流程
```
系统拆解 → Markdown底稿 → SVG图示 → fact-checker校验 → HTML成品
```

## 何时使用

当用户要做以下任一类任务时使用本技能：

- 智能硬件整机拆解、原理分析、系统架构重建
- 生成整机框图、运行流程图、模块职责拆解
- 关键器件识别、器件选型判断、替代方案比较
- 信号链、电源链、控制链、热路径、力传递路径分析
- 精度、响应速度、功耗、可靠性、良率、成本或供应链判断
- DFM/DFX、失效模式、校准策略、制造约束分析
- 竞品技术对比、平台方案差异、报告撰写

如果用户只要营销文案、品牌故事或纯市场观点，不要用本技能作为主技能。

## 标准工作流

### Phase 1: 明确任务边界

至少确认以下信息：

- 分析对象：单器件、模组、子系统还是整机
- 目标问题：原理、选型、失效、竞品、成本、报告成稿
- 输出粒度：快速判断、技术备忘录、正式报告
- 证据来源：本地文件、图片、链接、公开资料、联网研究
- 关键约束：体积、精度、响应、功耗、成本、认证、环境等级

信息不完整时，先列出必要假设，再继续。

### Phase 2: 系统拆解

默认按以下五条链路重建系统：

1. **能量链**：输入电源 -> 电源管理 -> 负载 -> 峰值/平均功耗
2. **信号链**：传感器/输入 -> AFE/ADC -> MCU/SoC -> 算法 -> 输出
3. **控制链**：采样 -> 状态判断 -> 控制律 -> 执行机构 -> 反馈
4. **数据链**：本地存储 / 无线 / 总线 / App / 云端
5. **结构链**：受力、散热、密封、装配、公差、防护、振动路径

然后把五条链路映射成整机框图里的模块，并回答：
- 系统启动后先运行什么
- 主控如何调度各模块
- 数据从哪里采集，在哪里处理，输出到哪里
- 哪些模块是核心功能模块，哪些是支撑模块
- 哪些模块之间强耦合，哪些可以平台复用

### Phase 3: 输出整机框图与运行流程

#### 整机模块框图

至少覆盖这些层：
- 电源与充电
- 传感与输入
- 主控与存储
- 通信连接
- 执行与输出
- 人机交互
- 结构与热管理
- App / 云端（若存在）

#### 运行流程

至少描述以下阶段：
1. 上电 / 唤醒
2. 自检 / 初始化
3. 采样与输入获取
4. 本地计算 / 滤波 / 状态判断
5. 控制执行或结果输出
6. 本地显示 / 通信上传 / 用户反馈
7. 待机、异常处理或低功耗切换

### Phase 4: 深度技术分析

至少覆盖以下问题：

#### 原理与架构
- 关键输入、输出、能量或信号如何转换
- 整机由哪些模块组成，各模块边界是什么
- 哪个模块负责感知、计算、控制、通信、显示、供电
- 模块之间通过什么接口或总线协作
- 上电到稳定运行的主流程是什么
- 关键模块如何分工，瓶颈位于哪一层
- 指标是由哪几个环节共同决定的

#### 器件与方案
- 主控、传感器、AFE、驱动、PMIC、无线、存储、执行器是什么类型
- 为什么会采用这一类方案，而不是相邻替代方案
- 哪些地方可能受制于算力、带宽、噪声、热、尺寸或认证

#### 结构与制造
- 力、热、振动、EMI、液体、灰尘如何影响系统
- 材料、壁厚、公差、固定方式、密封方式、装配顺序是否合理
- 哪些细节直接决定良率、校准复杂度和售后风险

#### 算法与固件
- 采样、滤波、校准、补偿、融合、控制、诊断流程是什么
- 哪些参数决定精度、稳定性、响应速度、续航或体验
- 固件升级、故障保护、异常恢复是否是系统短板

#### 成本与供应链
- 电子 BOM、结构 BOM、制造、测试、标定、认证分别由什么构成
- 哪些器件是单点风险，哪些工序是隐性成本
- 降本是否会伤害精度、可靠性或一致性

## 执行原则

1. **先定义问题，再分析** - 不先问"它是什么"，先问"用户究竟要判断什么"
2. **先证据，后推断** - 优先使用用户提供资料、本地文件、官方规格书、认证信息、专利、拆机资料
3. **先系统，后器件** - 先建立整机或模组架构，再落到芯片、传感器、结构细节
4. **先链路，后参数** - 沿着能量链、信号链、控制链、数据链、结构链拆解
5. **整机框图优先** - 先说明整机由哪些模块组成、模块之间如何连接
6. **先底稿，后展示** - Markdown 负责沉淀事实；HTML 负责高保真呈现；SVG 负责可复用图示
7. **明确结论等级** - 每个关键判断都要区分为：已确认事实 / 高概率工程推断 / 待验证假设
8. **不凭空指定具体型号** - 没有直接证据时，不要武断给出具体芯片型号、工艺型号、材料牌号

## 证据分级

默认按以下优先级组织证据：

1. 官方规格书、设计文档、认证数据库、原厂资料
2. 用户提供文件、实拍图、拆机图、原理图、测试记录
3. 权威评测、工程论文、专利、开发板资料
4. 二手拆机、论坛讨论、公开 BOM 猜测

输出时尽量标注证据强度。若只能做工程推断，应写明依据与反例可能性。

## 注意事项

1. **显式调用** - 本技能仅响应显式调用
2. **区分结论等级** - 必须区分已确认事实、高概率推断、待验证假设
3. **参数带单位** - 所有参数必须带单位，必要时补测试条件
4. **整机视角** - 必须说清"谁负责什么、谁和谁交互、系统如何跑起来"
5. **Markdown与HTML同源** - 不要出现结论不一致
6. **必须校验** - 所有关键数据必须经过 fact-checker 验证
