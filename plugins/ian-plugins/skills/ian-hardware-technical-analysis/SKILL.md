# 智能硬件整机技术分析

把智能硬件问题拆成可验证的整机工程问题，而不是停留在参数罗列或零散器件描述。优先重建整机框图、运行流程和模块职责，再下钻到器件、结构和算法细节。

## 何时使用

当用户要做以下任一类任务时使用本技能：

- 智能硬件整机拆解、原理分析、系统架构重建
- 生成整机框图、运行流程图、模块职责拆解
- 关键器件识别、器件选型判断、替代方案比较
- 信号链、电源链、控制链、热路径、力传递路径分析
- 精度、响应速度、功耗、可靠性、良率、成本或供应链判断
- DFM/DFX、失效模式、校准策略、制造约束分析
- 竞品技术对比、平台方案差异、报告撰写

如果用户只要营销文案、品牌故事或纯市场观点，不要用本技能作为主技能。

## 默认交付要求

只要对象是智能硬件整机或带嵌入式控制的系统，默认必须给出以下三项：

1. 整机模块框图
2. 整机运行流程
3. 各模块职责与边界

如果证据不足，仍然要给出“已确认部分 + 推断部分”，并明确哪些模块关系尚未证实。

## 输出介质策略

默认采用双层交付，而不是在 Markdown 和 HTML 之间二选一：

1. `Markdown` 作为技术底稿
2. `HTML` 作为展示型成品
3. `SVG` 作为整机图示资产

执行规则：

- 默认先产出结构化 Markdown，再生成 HTML
- 只要任务包含整机拆解、汇报展示、演示、评审材料或核心技术讲解，默认追加 HTML
- 只要任务涉及整机架构，默认生成至少 2 张 SVG：
  - 整机模块框图
  - 整机运行流程图
- 如果用户强调“演示感”“动态展示”“原理动画”，优先做 HTML，不要只交 Markdown
- 如果任务只是快速研究、内部笔记或需要频繁人工改写，可以只交 Markdown，但仍建议保留图示草稿

不要把 HTML 当作唯一事实来源。事实、参数、结论、证据分级应先落在 Markdown，再映射到 HTML。

## Claude Code 执行原则

1. 先定义问题，再分析。
   不先问“它是什么”，先问“用户究竟要判断什么”。

2. 先证据，后推断。
   优先使用用户提供资料、本地文件、官方规格书、认证信息、专利、拆机资料、评测、公开 BOM 线索。

3. 先系统，后器件。
   先建立整机或模组架构，再落到芯片、传感器、结构细节，避免局部结论脱离系统边界。

4. 先链路，后参数。
   沿着能量链、信号链、控制链、数据链、结构链拆解，参数只作为支撑，不单独堆砌。

5. 整机框图优先。
   对智能硬件，先说明整机由哪些模块组成、模块之间如何连接、数据和能量如何流动、系统如何从上电运行到输出结果。

6. 先底稿，后展示。
   Markdown 负责沉淀事实、参数、证据和结论；HTML 负责高保真呈现；SVG 负责可复用图示。

7. 明确结论等级。
   每个关键判断都要区分为：
   - 已确认事实
   - 高概率工程推断
   - 待验证假设

8. 不凭空指定具体型号。
   没有直接证据时，不要武断给出具体芯片型号、工艺型号、材料牌号或 BOM 数字。

## 任务分流

先判断用户任务属于哪一类，并调整输出重心：

| 任务类型 | 重点输出 |
|---|---|
| 原理拆解 / reverse engineering | 整机框图、运行流程、关键物理机制、核心器件作用 |
| 整机架构分析 | 模块划分、模块职责、接口关系、主控与外设协作 |
| 器件选型 / 方案比较 | 指标分解、候选方案、权衡逻辑、替代风险 |
| 故障 / 可靠性分析 | 失效模式、根因链、边界条件、验证方法 |
| 成本 / cost-down | BOM 拆分、制造与校准成本、替代件与降本风险 |
| 结构 / DFM / DFX | 材料、公差、装配、密封、散热、EMI/EMC、良率影响 |
| 竞品技术对比 | 模块级差异、性能兑现路径、架构取舍、供应链策略 |
| 报告生成 | Markdown 底稿、HTML 展示稿、SVG 图示、结论与建议 |

如果用户没有指定重点，默认覆盖顺序是：整机框图、运行流程、模块分工、关键链路、器件、结构、算法、可靠性、成本、竞品。

## 标准工作流

### 1. 明确任务边界

至少确认以下信息：

- 分析对象：单器件、模组、子系统还是整机
- 目标问题：原理、选型、失效、竞品、成本、报告成稿
- 输出粒度：快速判断、技术备忘录、正式报告
- 输出介质：仅 Markdown，还是 Markdown + HTML + SVG
- 证据来源：本地文件、图片、链接、公开资料、联网研究
- 关键约束：体积、精度、响应、功耗、成本、认证、环境等级

信息不完整时，先列出必要假设，再继续。

### 2. 先做系统拆解

默认按以下五条链路重建系统：

1. 能量链：输入电源 -> 电源管理 -> 负载 -> 峰值/平均功耗
2. 信号链：传感器/输入 -> AFE/ADC -> MCU/SoC -> 算法 -> 输出
3. 控制链：采样 -> 状态判断 -> 控制律 -> 执行机构 -> 反馈
4. 数据链：本地存储 / 无线 / 总线 / App / 云端
5. 结构链：受力、散热、密封、装配、公差、防护、振动路径

然后把五条链路映射成整机框图里的模块，并回答：

- 系统启动后先运行什么
- 主控如何调度各模块
- 数据从哪里采集，在哪里处理，输出到哪里
- 哪些模块是核心功能模块，哪些是支撑模块
- 哪些模块之间强耦合，哪些可以平台复用

如果是纯机械件，重点放在结构链和制造链；如果是纯电子模组，重点放在能量链、信号链和控制链。

### 3. 先输出整机框图与运行流程

对智能硬件整机，默认至少重建下面两类内容：

#### 整机模块框图

至少覆盖这些层：

- 电源与充电
- 传感与输入
- 主控与存储
- 通信连接
- 执行与输出
- 人机交互
- 结构与热管理
- App / 云端（若存在）

#### 运行流程

至少描述以下阶段：

1. 上电 / 唤醒
2. 自检 / 初始化
3. 采样与输入获取
4. 本地计算 / 滤波 / 状态判断
5. 控制执行或结果输出
6. 本地显示 / 通信上传 / 用户反馈
7. 待机、异常处理或低功耗切换

如果用户要求深入，继续说明每个阶段由哪些模块协作完成。

### 4. 组织交付文件

除非用户明确不要，否则按以下顺序组织产物：

1. `report.md`
2. `report.html`
3. `svg/overview-block-diagram.svg`
4. `svg/system-flow.svg`

若任务更深，可继续增加：

- `svg/signal-chain.svg`
- `svg/power-chain.svg`
- `svg/control-loop.svg`
- `svg/core-tech-principle.svg`

要求如下：

- Markdown 是事实主稿，结构稳定、易编辑、易 diff
- HTML 不应重新发明内容，而应增强可读性、层次和演示感
- SVG 应尽量独立成文件，以便高清显示、局部复用和后续动画
- 动画优先使用 HTML + CSS / SVG 动画，不要把关键信息只藏在动效里

### 5. 归类对象并切换关注点

| 对象类型 | 默认关注点 |
|---|---|
| 传感器 / 测量类 | 物理原理、误差源、温漂、标定、AFE、滤波 |
| 控制器 / 执行器类 | 控制周期、实时性、驱动能力、反馈闭环、热管理 |
| 电源 / 供电类 | 转换效率、纹波、瞬态响应、保护策略、续航 |
| 通信 / 连接类 | 协议栈、射频/布线约束、时延、吞吐、兼容性 |
| 机械结构件 / 模组 | 力传递、刚度、公差、疲劳、密封、装配与工艺 |
| 整机设备 | 子系统耦合、系统瓶颈、校准流程、制造一致性 |

### 6. 做深度技术分析

至少覆盖以下问题：

#### 原理与架构

- 关键输入、输出、能量或信号如何转换
- 整机由哪些模块组成，各模块边界是什么
- 哪个模块负责感知、计算、控制、通信、显示、供电
- 模块之间通过什么接口或总线协作
- 上电到稳定运行的主流程是什么
- 关键模块如何分工，瓶颈位于哪一层
- 指标是由哪几个环节共同决定的

#### 器件与方案

- 主控、传感器、AFE、驱动、PMIC、无线、存储、执行器是什么类型
- 为什么会采用这一类方案，而不是相邻替代方案
- 哪些地方可能受制于算力、带宽、噪声、热、尺寸或认证

#### 结构与制造

- 力、热、振动、EMI、液体、灰尘如何影响系统
- 材料、壁厚、公差、固定方式、密封方式、装配顺序是否合理
- 哪些细节直接决定良率、校准复杂度和售后风险

#### 算法与固件

- 采样、滤波、校准、补偿、融合、控制、诊断流程是什么
- 哪些参数决定精度、稳定性、响应速度、续航或体验
- 固件升级、故障保护、异常恢复是否是系统短板

#### 成本与供应链

- 电子 BOM、结构 BOM、制造、测试、标定、认证分别由什么构成
- 哪些器件是单点风险，哪些工序是隐性成本
- 降本是否会伤害精度、可靠性或一致性

### 7. 输出工程判断

结论必须回答：

- 该方案为什么成立
- 该方案的上限和短板分别由什么决定
- 哪些指标可能是“规格好看但场景兑现有限”
- 哪些问题必须通过补测、拆机、原理图或实验才能确认

## 证据分级

默认按以下优先级组织证据：

1. 官方规格书、设计文档、认证数据库、原厂资料
2. 用户提供文件、实拍图、拆机图、原理图、测试记录
3. 权威评测、工程论文、专利、开发板资料
4. 二手拆机、论坛讨论、公开 BOM 猜测

输出时尽量标注证据强度。若只能做工程推断，应写明依据与反例可能性。

## 参考资料加载规则

需要参数维度、系统拆解框架、误差来源、成本与可靠性分析角度时，读取 [references/technical-parameters.md](references/technical-parameters.md)。

不要整份机械照搬参考文件。只抽取与当前对象最相关的部分。

## 输出模板

### 快速分析

适合单轮问答或初筛：

- 对象与任务目标
- 整机框图（文字版即可）
- 运行流程（按阶段）
- 系统拆解（五条链路中最关键的 2-3 条）
- 核心技术判断
- 主要风险 / 短板
- 待验证项

### 技术备忘录

适合方案讨论、设计评审或内部研究：

1. 分析对象与边界
2. 整机框图与模块职责
3. 运行流程与模块协作
4. 系统架构与关键链路
5. 关键器件 / 结构 / 算法判断
6. 性能、功耗、可靠性、成本分析
7. 风险列表与验证建议

### 完整技术报告

适合正式交付件：

1. 产品概述与场景定义
2. 关键需求与指标拆解
3. 整机框图与模块分工
4. 整机运行流程与状态转换
5. 系统架构与关键链路分析
6. 关键器件与选型判断
7. 结构、制造与 DFM/DFX 分析
8. 算法 / 固件 / 控制策略
9. 性能、功耗、可靠性与环境适应性
10. 成本、供应链与量产风险
11. 竞品技术对比
12. 结论、风险与下一步验证建议

### 网页化交付

如果用户要求 HTML 或展示型报告：

- 以 Markdown 主稿为基础映射，不要写成另一套内容
- 保持工程评审材料风格，而不是营销落地页
- 默认内嵌或引用 SVG 图示，而不是使用低清截图
- 优先使用系统框图、运行流程图、参数对比表、风险表、结论摘要
- 可以加入分步高亮、链路点亮、模块 hover、分层展开等轻量动画
- 动画只做解释增强，不替代正文和结论
- 如果当前环境存在前端设计类 skill，再调用它生成页面

### 推荐组合

默认推荐：

- `report.md`：完整技术正文
- `report.html`：展示型成品
- `svg/*.svg`：图示资产

若时间有限，优先级如下：

1. `report.md`
2. `overview-block-diagram.svg`
3. `system-flow.svg`
4. `report.html`

## 写作要求

- 区分事实、推断、假设，不混写
- 参数必须带单位；必要时补测试条件和量纲
- 公式只在有助于判断时使用，并写清前提
- 不只写“更强/更快/更准”，要说明由什么机制带来
- 整机分析必须说清“谁负责什么、谁和谁交互、系统如何跑起来”
- Markdown 与 HTML 内容必须同源，不要出现结论不一致
- SVG 图示要能独立阅读，模块名、箭头方向、链路含义要清楚
- 对竞品差异，优先解释架构差异，而不是只比参数
- 遇到缺数据场景，先告诉用户最关键还缺什么资料
